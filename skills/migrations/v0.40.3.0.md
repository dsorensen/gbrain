# v0.40.3.0 Migration: Contextual Retrieval

**TL;DR:** Anthropic's published contextual retrieval method ships in
gbrain as a three-tier wrapper ladder gated by `search.mode`. Every
markdown page re-embeds through the new wrapper path on upgrade —
title-only at the `balanced` tier (free at runtime), per-chunk Haiku
synopsis at the `tokenmax` tier (premium, opt-in via `gbrain config
set search.mode tokenmax`). The mechanical side runs automatically
via `gbrain upgrade`; this file documents what to verify post-upgrade.

## What changed

**Tier ladder (search.mode-gated):**
- `conservative` → no wrapper (minimum surface)
- `balanced` (default) → title-only `<context>{title}</context>` prefix
- `tokenmax` → per-chunk Haiku synopsis (Anthropic's exact published method)

**Schema (migration v81):**
- `pages.contextual_retrieval_mode TEXT NULL` — tier the page was last embedded under
- `pages.corpus_generation TEXT NULL` — composite hash for cache invalidation
- `sources.contextual_retrieval_mode TEXT NULL` — per-source override (CLI-write-only)
- `sources.trust_frontmatter_overrides BOOLEAN DEFAULT FALSE` — mount-frontmatter trust gate
- `query_cache.page_generations JSONB DEFAULT '{}'` — D27 P1-5 invalidation contract foundation

**Chunker version 2 → 3:** signals the post-upgrade reembed sweep that
every markdown page needs to be re-embedded through the new wrapper.
Chunk boundaries themselves are unchanged.

**KNOBS_HASH_VERSION 3 → 4:** query cache rows from prior CR-mode-less
brains naturally miss; one-time hit-rate dip clears within the cache TTL.

**Race fix folded in (D24):** the long-standing v0.35.x NULL→non-NULL
upsert race between concurrent embed workers is closed. Two writers
racing on the same chunk now let the fresher `embedded_at` win.

## Mechanical side (handled by `gbrain upgrade`)

1. **Migration v81 applies via `gbrain apply-migrations --yes`** —
   tail of `runPostUpgrade()` already triggers this; no manual action.
2. **Post-upgrade reembed prompt fires automatically** with the v0.40.3.0
   contextual-retrieval explanation appended to the chunker-bump cost
   summary. 10-second Ctrl-C grace window; `GBRAIN_NO_REEMBED=1`
   bypasses if you're upgrading on a brain you don't want to re-embed
   right now (CI, ephemeral brains, etc.).
3. **The post-upgrade reembed runs `gbrain reindex --markdown`** which
   walks pages where `chunker_version < 3 OR contextual_retrieval_mode
   IS NULL` — the v0.40.3.0 sweep predicate that catches both v2-chunker
   pages AND pages where the new column hasn't been stamped yet.

## Verify the outcome

```bash
gbrain doctor --json                     # contextual_retrieval_coverage should be 'ok'
gbrain search "any query you'd normally run"  # confirm queries return results
gbrain stats                             # brain_score
```

If `gbrain doctor` warns about `contextual_retrieval_coverage`:

```bash
gbrain reindex --markdown                # re-run the sweep (idempotent)
gbrain doctor                            # should clear after sweep completes
```

## To opt up to the per-chunk Haiku synopsis tier

The `tokenmax` tier runs a Haiku call per chunk to generate a chunk-
specific synopsis (Anthropic's published method). One-time backfill cost:

| Brain size | Backfill cost (Haiku 4.5) | Wall-clock at default Anthropic 50 RPM |
|-----------|---------------------------|-----------------------------------------|
| 10K pages | ~$1-5 | ~17 hours |
| 50K pages | ~$5-25 | ~85 hours (3.5 days) |

```bash
gbrain config set search.mode tokenmax
gbrain reindex --markdown    # submits contextual_reindex_per_chunk Minion jobs
gbrain jobs list --queue default --status active  # monitor backfill
```

If you have higher Anthropic Haiku quota, override the rate cap:
```bash
GBRAIN_CONTEXTUAL_HAIKU_RPM=200 gbrain jobs work --queue default
```

## To opt out (soft kill switch)

If post-upgrade search quality regresses, flip the soft kill switch.
Wrapped vectors stay in the DB but new embeds skip the wrapper.
Existing wrapped vectors keep serving queries via cosine similarity
(asymmetric retrieval is preserved):

```bash
gbrain config set search.contextual_retrieval_disabled true
```

To re-enable:
```bash
gbrain config unset search.contextual_retrieval_disabled
```

## Per-page surgical control

Power users can override the global tier for one specific page via YAML
frontmatter:

```yaml
---
title: "My long-form essay"
contextual_retrieval: per_chunk_synopsis
---
```

Valid values: `none`, `title`, `per_chunk_synopsis`. Typos surface as
SyncFailure entries in `gbrain doctor`'s `sync_failures` check with
the raw bad value for debugging.

**Security note (D15):** per-page frontmatter overrides from MOUNTED
sources (team brains, third-party brains) are REJECTED by default to
prevent a third-party brain from draining your Haiku quota silently.
Future v0.40.4+ ships `gbrain mounts trust-frontmatter <source-id>`
to opt-in per-source. For v0.40.3.0, the host source (id='default')
is always trusted; mounted-source frontmatter overrides fall through
to source/global.
